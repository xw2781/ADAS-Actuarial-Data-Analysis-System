A human-readable map of how the in-app ArcBot chat gathers app context, builds the Codex prompt, launches each CLI turn, applies edits, and reports progress back to the UI.

One-Screen Summary

1 User Sends Request The chat UI records the message, starts a work card, and asks the active tab for app context if enabled.
2 Host Prepares Context Electron validates the active JSON target and creates a local editable temp copy for Edit Mode.
3 Prompt Is Assembled The prompt includes safety rules, mode rules, paths, active app context, attachments, and recent chat history.
4 Codex CLI Runs ArcBot tries the warm Codex session first, then falls back to one-shot codex exec with stdin prompt input.
5 Result Returns The host validates edits, backs up originals, applies allowed changes, and streams user-facing progress to the chat.

Important Concepts

App Context Temp JSON Copy Codex Sandbox Host Validation
  • App Context is the active tab summary and JSON-backed data supplied by the app UI.
  • Temp JSON Copy is the only file Codex should edit in Edit Mode.
  • Host Validation is the actual safety boundary; Codex suggestions are checked before touching project files.
  • Prompt Rules guide the model, but file access is enforced by the host and sandbox choices.

Prompt Assembly

Where the prompt is built

The current prompt builder is buildAssistantPrompt(...) in frontend/electron/main.js. Each request builds a new prompt from runtime state.

const rawPrompt = buildAssistantPrompt(
  payload?.messages,
  mode,
  roots.projectRoot,
  codexCwd,
  roots.networkRoot,
  activeContext,
  activeJson,
  editSession,
  attachments
);

const prompt = clampCodexPrompt(rawPrompt);
What goes into each prompt
Prompt section Purpose
You are ArcBot... Sets identity and tone for the in-app assistant.
Current mode Separates Review Mode and Edit Mode behavior.
Current project folder and CLI working folder Explains where the project is and where Codex was launched.
Mode instructions Read-only restrictions for Review Mode; temp-copy edit rules and JSON-only response contract for Edit Mode.
Active page context Passes active tab metadata, target path, file state, and UI-derived context.
Active local JSON-backed data Either references the temp file name or embeds active JSON when no temp edit session exists.
Attached file context Includes user-attached text-like files, limited to a bounded size.
Conversation Includes the most recent chat messages so the request has session context.
Current mode-specific prompt rules

Edit Mode: Codex may edit only the active JSON-backed temp copy in the CLI working folder. The final visible answer must be inside a single JSON object, for example {"action":"edited","reply":"..."}.

Review Mode: Codex must not edit files, change settings, install packages, run destructive commands, commit, or push. It returns a concise answer.

Prompt Size and Context Window

  • clampCodexPrompt(...) caps the prompt at about 200,000 characters.
  • The UI estimates token usage with a rough chars / 4 calculation.
  • The displayed context window default is 200,000 tokens, controlled by ARCRHO_CODEX_ASSISTANT_CONTEXT_WINDOW_TOKENS.
  • Attachments are limited to the first five text-like files, each capped before prompt insertion.

Codex CLI Interaction Model

Warm Session Path

ArcBot first tries a warm Codex app-server turn through runCodexWarmTurn(...).

This is intended to behave closer to an ongoing Codex app session and can stream activity back to the UI.

Fallback CLI Path

If the warm session fails, ArcBot falls back to one-shot codex exec.

codex exec --ephemeral --color never --sandbox ... --skip-git-repo-check --cd ...

Prompt Delivery

The prompt is passed through stdin, not as a positional command argument.

This avoids Windows quoting failures such as unexpected argument errors.

Edit Safety Model

  1. The active tab returns a JSON-backed target path and active JSON payload.
  2. The host validates whether the target can be edited under allowed rules.
  3. The host creates a session folder under the local ArcBot workspace.
  4. Codex edits only the temp copy, not the true server file.
  5. The host parses and validates the result.
  6. The host backs up the original to a history location before applying.
  7. The host applies the temp copy to the real target only after validation.
Key point: prompt instructions help, but the important protection is that the host owns backup, validation, and final apply.

UI Event and Progress Flow

What the user sees
  • A Thinking ... bubble appears immediately.
  • The progress card opens while running and collapses after completion.
  • User-facing progress is summarized in the work card.
  • Technical stdout, stderr, raw host messages, and detailed logs stay in the debug drawer.
  • The final assistant reply can render bullets, bold text, inline code, and code blocks.
How progress gets streamed

The host sends events such as activity, stdout, stderr, usage, and context. The renderer maps those events into user-facing progress bullets and updates the progress card incrementally.

What is stored locally
  • Chat sessions and messages are stored in local JSON under Electron user data.
  • Temp JSON edit copies are stored under the local ArcBot sessions folder.
  • Latest edit metadata is stored so ArcBot can revert the latest validated edit when safe.

What Is Good To Move Out Of main.js

  • Prompt text templates: base, edit-mode, and review-mode text.
  • Domain context docs: DFM JSON layout, cell status meanings, label rules, and examples.
  • Display wording maps: user-facing progress phrases, if they grow large.

What Should Stay Enforced In Code

  • Sandbox choice: read-only vs workspace-write.
  • Allowed target validation: only configured app/server targets or active notebook exceptions.
  • Backup and revert rules: copy original before apply, refuse unsafe revert.
  • JSON parsing and final apply: never rely only on model text to change real files.

Question Current answer Possible next improvement
Can a human review the prompt? Yes, but it is embedded in main.js. Move prompt templates into Markdown files loaded by the host.
Can Codex edit real files directly? Design intent is no for ArcBot app-data edits; Codex edits the temp copy, host applies. Keep prompt rules and validation aligned, and add explicit tests around target validation.
Can users understand what is happening? Progress card shows user-facing steps; debug drawer holds technical logs. Continue mapping raw Codex/tool messages to concise app-specific updates.
Can context grow too large? Yes. There is prompt clamping and token estimation, but no semantic retrieval yet. Add structured context packs or lazy file references for very large datasets.
